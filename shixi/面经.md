**问题：Agent 相比传统 Workflow 有什么本质区别？在真实的业务场景中应该如何权衡与选型？**

**回答：** 传统 Workflow 就像按既定轨道运行的高铁，依靠预设的规则与条件分支驱动，具备极高的确定性和绝对的控制权，适用于银行审批、标准数据清洗等流程固定且对错误零容忍的兜底场景；而Agent则像自动驾驶汽车，以大模型为大脑进行动态规划和工具调用，具备极强的自主性与灵活性，适合应对复杂 Text2SQL、开放域分析等路径未知、输入高度非结构化的探索型任务。在当前大厂真实的工业落地实践中，最高效的架构通常是“Agentic Workflow（智能工作流）”，即在宏观主干流程上采用 Workflow 进行坚固的强管控兜底，而在需要复杂逻辑推理的微观节点上挂载 Agent 进行弹性探索，从而实现系统稳定性与 AI 泛化能力的极致平衡。

**问题：目前主流的 Agent 开发范式有哪些？分别适用于什么业务场景？**
**回答：** 主流的 Agent 开发范式主要有三种：

1. **ReAct（边想边做）：** 遵循 `思考-行动-观察` 的微循环。适用于短链路、需要与环境高频交互试错的场景（如简单的 Text2SQL 或网页检索）。但应对复杂任务易陷入死循环。
    
2. **Plan-and-Execute（先规划后执行）：** 将任务拆解为子目标列表再逐一执行。适用于长链路、目标明确的复杂任务（如自动化研报生成），能有效防止大模型在中途迷失方向。
    
3. **Multi-Agent（多智能体协作）：** 引入多角色设定进行协作或对抗（如 Coder 与 Reviewer）。适用于极其复杂、需要多视角校验和纠错的工程级任务（如全流程软件生成）。
    

**问题：在 Agent 运行中，历史记录和工具返回导致上下文过长（Token 爆炸），怎么解决？**

**回答：** 解决上下文溢出需要从“记忆调度”和“底层基建”双管齐下：

1. **记忆工程优化（核心）：** 抛弃简单的滑动窗口截断。采用“RAG 检索”精准召回相关上下文；或采用更高阶的“状态图谱更新（Memory Management）”方案，后台使用独立 Agent 抽取历史对话中的实体状态，执行软删除与覆盖，每次只向主干链路挂载最新的、高信息熵的 Entity 状态，从根源遏制上下文线性增长。
    
2. **Infra 层优化（基建）：** 结合 vLLM 等高性能推理框架，开启 **Prefix Caching（前缀缓存）**，将长串的 System Prompt 和工具描述固化在 KV Cache 中，大幅降低长文本场景下的算力消耗和首包延迟。
