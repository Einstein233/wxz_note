- **狙击点：** 当被问及上下文优化的效果时，回答“主要依靠产品经理的反馈”。这在大厂是极其忌讳的，大厂工程演进必须由客观数据指标驱动。

    - **业务指标（自动化评测）：** 建立包含核心 Ground Truth 的测试集，每次修改摘要逻辑后，进行自动化 RAG 召回测试，计算 **召回命中率（Hit Rate）**，用客观数据指导迭代。
### 🔥 核心拷问 2：流式音频技术栈“太老旧”

- **狙击点：** 简历中的双向流式语音后端，被面试官指出架构老旧、有更好的替代方案。

- **✅ 迭代后的“大厂级”解法（前沿演进视野）：**
    
    - **认知对齐：** 承认级联架构的延迟天花板。
        
    - **演进方向：** 下一代极致拟人与极低延迟（<300ms）的架构，必须演进到 **端到端原生音频流模型（End-to-End Audio Native）**，直接让大模型接收并输出音频，跳过中间文本转换。

### 🔥 核心拷问 3：Python 为什么不能实现真正的多进程/多线程？

- **狙击点：** 考察对 Python 内存管理与 GIL 锁机制的精确理解。
    
- **❌ 我的原始漏洞：** 概念张冠李戴。误以为 Python 不能实现真正的多进程，且将 GIL 的限制范围搞错。
    
- **✅ 迭代后的“大厂级”解法（精准区分资源与调度）：**
    
    - **澄清事实：** Python **完全可以**实现真正的多进程（`multiprocessing`），因为进程是资源分配的最小单位，每个进程有独立的内存和解释器，可实现**真并行（Parallelism）**。
        
    - **直击痛点（GIL 锁）：** Python 无法实现的是真正的**多线程并行**。因为线程是调度单位，共享内存。受限于 CPython 的全局解释器锁（GIL），同一时刻一个进程内只有一个线程能执行 CPU 计算。
        
    - **应用场景结论：** 计算密集型（如大模型本地推理）用多进程；I/O 密集型（如网络请求）用多线程或协程。
        

### 🔥 核心拷问 4：FastAPI 是如何处理高并发的？背后原理是什么？

- **狙击点：** 考察对 FastAPI 底层机制（Asyncio）的理解，以及“并发”与“并行”的概念区分。
    
- **❌ 我的原始漏洞：** 停留在“调包侠”视角，误以为 FastAPI 是通过模拟多线程并行来处理并发。
    
- **✅ 迭代后的“大厂级”解法（异步非阻塞与事件循环）：**
    
    - **核心机制：** FastAPI 底层（基于 ASGI 和 `uvloop`）采用的是**单线程下的异步非阻塞 I/O (Async I/O)**。
        
    - **事件驱动模型 (Event Loop)：** 当遇到 `await`（如等待大模型 API 响应）这类 I/O 耗时操作时，当前线程**绝对不会阻塞等待**，而是将任务挂起，事件循环立刻切换去处理下一个网络请求。
        
    - **概念升华：** 它实现的是**高并发 (Concurrency)**——通过榨干单线程的等待时间来交替处理海量请求，而不是**并行 (Parallelism)**——物理上的同时执行。
