### 1. 核心架构与技术栈选型

这个项目不需要前端页面（UI 不重要），我们的所有火力全部集中在后端并发吞吐和底层监控上。

- **底层环境：** 实验室台式机（必须安装 Ubuntu 20.04 或 22.04，配置好 Nvidia 驱动和 CUDA 12.x）。
    
- **开源模型：** 选择百川（Baichuan2-7B）、千问（Qwen1.5-7B）或 Llama-3-8B（这类 7B/8B 级别的模型，一张 24G 显存的 RTX 3090/4090 就能完美跑起 vLLM）。
    
- **推理引擎（核心）：** 抛弃 Ollama，直接全源码编译或通过 pip 安装 **vLLM**。
    
- **网关接入层：** 使用 **FastAPI (Python)** 编写一个反向代理网关，模拟真实大厂的 API 鉴权（Token 校验）和限流（Rate Limiting）。
    
- **高并发压测工具：** 使用 **Locust** 或 **JMeter**。这是项目出彩的关键，我们要用它模拟 100 个用户同时疯狂发送长文本请求。
    
- **可观测性看板（绝杀项）：** 部署 **Prometheus + Grafana**。抓取 vLLM 暴露的底层 metrics（如 KV Cache 占用率、排队请求数），做成炫酷的实时监控大屏。
    

### 2. 项目推进的“四步走”战役

- **第一步：裸跑 vLLM 与基准测试。** 在 Linux 上跑通 vLLM 的 API Server 模式。写一个简单的 Python 脚本，测试在单并发下，首字延迟（TTFB）是多少，整体生成速度是多少。
    
- **第二步：构建 API 网关。** 不要让外部直接访问 vLLM。你用 FastAPI 写一个中间层，接收请求后转发给 vLLM。在这一层加上时间戳记录，精确计算每个请求在网关层排队的时间。
    
- **第三步：火力全开的极限压测。** 写一段 Locust 压测脚本，模拟 50、100、200 个并发用户。这里你要亲眼观察：当并发达到多少时，显存会耗尽？响应延迟会从几十毫秒飙升到几秒？
    
- **第四步：点亮“可观测性”技能树。** vLLM 原生支持 Prometheus 格式的数据指标暴露。把它接入 Grafana 面板。在这个面板上，你要能实时看到“PagedAttention 的内存页使用率”和“Continuous Batching 的批次大小”。
