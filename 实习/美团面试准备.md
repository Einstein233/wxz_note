
### 1、SQL八股
**如何防止SQL注入：**
	1、SQL注入就是，代码里直接拼接了字符串。因此只要构造特定的输入，就会导致执行一些不合法的数据库操作；
	2、如何防止SQL注入：
	常见方法**参数化查询**，先给数据库发送SQL的“骨架”，再发送填充骨架的“数据”
	**使用ORM框架**底层还是参数化查询；
	3、我们的项目中是如何避免SQL注入的：
	**数据库权限隔离**我们使用MCP来执行SQL的数据库连接配置，使用的不是root账号，我们使用的是一个只读账号，这个刚好不能进行一些危险操作的权限。
	**鉴权层**对这条字符串进行了静态分析，通过正则匹配来确定这个这个SQL命令的操作类型，只允许SELECT类型的语句执行；
	**应用层白名单机制**我们在挂在表格的时候，只挂载其后相关的数据表，因此模型不能查询密码表等
	同时基于提示词的设计，可以很大概率的避免agent生成的SQL命令出现差错；

**多表联查，基本的区别**：
INNER JOIN （内连接，只寻找严格的交集）和LEFT JOIN（左连接，意思是以左边的表为准，左边全要，右侧的缺失值使用NULL来填充）核心区别在于未匹配的数据如何处理。
**GROUP BY和聚合函数**
**概念**：`GROUP BY` 就是“分类汇总”。比如把订单按“城市”分组，然后算出每个城市的订单总数（配合 `COUNT()` 或 `SUM()` 等聚合函数使用）。

**高频问题**：**`WHERE` 和 `HAVING` 有什么区别？**
**实战应对话术**：“它们都是用来过滤数据的，但**执行时机不同**。`WHERE` 是在数据**分组前**进行过滤，它不能跟着聚合函数（比如不能写 `WHERE SUM(价格) > 100`）；而 `HAVING` 是在数据**分组后**对汇总结果进行过滤。所以我通常用 `WHERE` 筛掉无效的原始数据，再用 `HAVING` 筛选出总金额达标的用户群体。”

**什么是 UNION？它在实际业务中用来干什么？** **A：** `UNION` 是 SQL 中的联合查询操作符。它就像“上下堆叠表格”，用来将两个或多个 `SELECT` 查询的结果**上下拼接**在一起，从而增加结果集的**行数（数据量）**。 _(对比记忆：`JOIN` 是“左右拼接”，用来增加结果集的列数/字段。)_

**使用 UNION 合并两张表时，有什么严格的判定规矩？** **A：** 记住三句话：“数量极度严格，类型相对宽容，名字完全无视。”
- **数量严格**：上下合并的 `SELECT` 语句，查询的字段（列）数量必须绝对一致，少一个直接报错。
- **类型宽容**：上下对应的列，数据类型不需要 100% 相同，只要能隐式兼容转换即可（如整数兼容字符串）。
- **名字无视**：`UNION` 只看字段的对应位置，不在乎列名叫什么。最终大表的列名默认采用第一个 `SELECT` 语句的列名。

**如果两张表字段数量不一样，但业务又强行需要 UNION 合并，该怎么处理？** **A：** 可以在字段较少的那个 `SELECT` 语句中，使用 `NULL` 或者常量字符串（例如 `'未知'`）作为**占位符**来凑数，从而强行满足 `UNION` 对列数相等的严格要求。

**UNION 和 UNION ALL 有什么本质区别？（🔥 超高频考点）** **A：** 核心区别在于**去重机制**和**性能**。
**`UNION`（带去重）**：合并数据后，会在底层自动执行**全局排序并去重**。这非常消耗 CPU 和内存性能，容易拖慢查询。
**`UNION ALL`（不去重）**：简单粗暴地将数据上下拼接，**不执行去重**。执行速度极快。
**实战原则**：默认优先使用 `UNION ALL`。除非业务严格要求数据库层面去重，否则尽量将数据拉出来，在业务代码层（如 Java 集合/Python 列表）去处理重复项，以减轻底层数据库的压力。

**MySQL 原生不支持 FULL OUTER JOIN（全外连接），怎么用 UNION 实现？** **A：** 全外连接需要“左表独有 + 右表独有 + 两边共有”的数据。在 MySQL 中的标准写法是：先用 `LEFT JOIN` 查出左边全量，再用 `RIGHT JOIN` 查出右边全量，最后用 **`UNION`** 把这两个结果合并。因为 `UNION` 自带去重功能，它会自动把两边重复的交集部分去重掉，完美实现全外连接的效果。****

### 2、python并发控制与性能排查
高并发控制：
	线程池，当请求量非常多的时候，就会导致显存溢出或者CPU资源占满导致无法处理多余请求而报错。
	因此需要使用线程池来控制最大的线程数量。例如，基于预设的最大线程数，或者基于剩余的硬件资源来控制可以建立的线程数量。
	控制最大可创建的线程数量，那么剩下的请求将会使用队列进行排队等待。如果使用queue虽然可以处理类似逻辑，但是在系统崩溃或者硬件重启之后会导致请求丢失。
	因此可以使用消息队列中间件来处理，消息队列专门负责存储并分发请求，请求被转发到服务器进行处理。完成削峰填谷的作用。
	绝对优势：持久化、分布式调度
	Redis本身不是调度引擎，但是在分布式调度架构中，扮演不可或缺的底层协调角色。例如：
	**作为状态共享的黑板**：分布式环境下的节点是无状态的。调度器可以通过redis来实时存储和读取集群中数千个节点的心跳状态、资源使用率，从而做出正确的派单决策。
	**作为任务缓冲队列（Task Queue）**：在遇到海量突发任务的时候，利用Redis的List或者Stream结构进行削峰填谷，平滑发给空闲的Worker节点。
	**作为防并发冲突的锁**：在多个调度器示例或者Worker节点可能会发生资源抢占时，利用redis的SETNX原句来实现分布式锁，确保同一个核心人物绝对不会被重复调度或重复执行。
	**缓存穿透、缓存击穿、缓存雪崩**：
	缓存穿透是，故意发送请求去查询一个根本不存在的数据，这样每次都缓存不命中，因此导向MySQL，数据库就会因为承受不住压力而导致阻塞挂掉。解决的方法是：可以即使查不到，也写入一个空值，并且设计一个很短的过期时间，这压根如果再查就可以直接redis进行返回了。或者使用布隆过滤器，也就是设计一个黑名单白名单，这样对于一些不存在的ID数据，直接进行拦截。
	缓存击穿是，如果此时缓存中没有数据，但同时来了很多请求，此时都发现缓存中没有数据，因此全部导向MySQL。这样单点高并发，直接将数据库卡死了。解决的方法是：使用Redis分布式锁SETNX。当一万个请求发现缓存没有命中的时候，不许直接查数据库，而是使用一个锁，只允许一个线程或者去查数据，查完写回redis这样后续的请求就不用再请求数据库了。
	缓存雪崩是，如果因为redis中的缓存同时过期，导致进入MySQL的请求瞬间增多，这样就会导致MySQL的服务崩溃。解决方法有两种；一是使用错峰过期，加入随机数。二是使用高可用集群例如k8s，这样即使一个redis服务崩溃，我们也可以使用另外的redis来补充上来。
性能排查：cProfile
python -m cProfile -s cumtime agent_task.py
使用上述的命令可以查询我们的代码文件的各个函数调用的时间消耗，
主要字段描述：
ncalls:调用次数，如果出现异常，可能是有逻辑bug
cuntime：累计耗时，是一个函数执行的时间，以及调用的所有子函数的时间。
tottime：净耗时，揪出CPU性能刺客的关键！只计算函数自己内部代码的执行花费时间。
可以得到的消息：
如果一个模块的cumtime很高，但是tottime很低，那说明这是一个IO瓶颈的程序，可以考虑异步编程或者多线程。
如果一个模块的tottime很高，那可能就是CPU瓶颈，是计算密集型的，可以考虑review进行重构。

2、使用redis的消息队列功能与请求拉取的设计来避免我们的LLM调用服务OOM。
redis是单独拉起的服务。然后将前端发送的请求，首先进入网关层，将请求添加元数据之后，添加到redis的消息队列中；
redis的内部，可以使用List这样初级的数据结构，或者Stream这样高级的数据结构，来避免在后端还没有处理完成请求的时候就将请求从缓存队列中删除；也可以使用持久化策略，开启AOF，来将队列中的数据刷到硬盘上。就算进程崩溃，重启最多丢失一小部分的数据；内存淘汰策略（Eviction Policy）拒绝淘汰，是当redis的缓存占满的时候，避免删掉旧的；当redis作为消息队列的时候，必须避免LRU。
