# 快手面试技术拷打问答

这次面试后半段，面试官对我简历里的几个技术点追问得很深。回头看，真正难的不是背概念，而是要把“我为什么这么设计、这个方案的边界在哪、下一步怎么演进”讲清楚。

**Q1：你做的双向流式语音后端，为什么会被说技术栈偏老？**

**A：** 因为我当时采用的是比较经典的级联方案：`ASR -> LLM -> TTS`。这个方案的优点是工程实现快、成本低、适合两个人的小团队先把业务闭环跑起来；但它的问题也很明显，一是链路长，延迟有天花板，二是语气、停顿、情绪这类信息在中间文本化过程中会丢失。  
如果从“能上线”角度看，这个方案没问题；但如果从更前沿的语音交互体验看，下一步应该演进到端到端的原生音频流模型，让模型直接接收和输出音频，而不是每一层都做一次格式转换。

**Q2：Python 到底能不能实现真正的多进程、多线程？**

**A：** 这个问题我当时被问得很细，核心是要把多进程和多线程分开说。  
Python 当然可以做真正的多进程，因为多个进程有独立内存和解释器，能利用多核实现真并行；受限更多的是 CPython 下的多线程 CPU 并行能力，因为 GIL 会让同一时刻只有一个线程执行 Python 字节码。  
结合我自己的项目也很好理解：我在压测系统里用 Python 协程和异步请求去打高并发，这更适合 I/O 密集场景；如果是本地推理、重计算任务，思路就应该往多进程或者服务拆分上走。

**Q3：FastAPI 为什么能扛高并发？你真的理解它的底层吗？**

**A：** 这个问题一下子就能区分“会用”和“理解原理”。  
我在项目里用 FastAPI 做过实时监控和压测后端，通过 WebSocket 推送 vLLM 的 GPU/KV Cache 指标，同时接收网页端下发的压测任务。它能支撑很多连接，不是因为“偷偷开了很多线程”，而是因为底层基于 ASGI 的异步非阻塞 I/O 模型。  
简单说就是：请求遇到网络等待、模型接口返回这类 I/O 阻塞点时，会通过 `await` 把执行权交还给事件循环，线程不会傻等，所以同一个服务进程可以把等待时间利用起来处理更多请求。这也是为什么它更适合我这种监控、推流、压测编排类场景。

如果让我总结这轮面试最大的感受，就是大厂不会只听你说“我做过”，而是更想听你讲清楚：这个方案为什么成立、为什么不够好、以及它该往哪里继续演进。
