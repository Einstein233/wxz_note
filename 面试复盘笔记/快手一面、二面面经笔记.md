## 快手二面经验
# 📝快手大模型 Agent 开发（二面·下半场）

- **📅 面试基本信息：** 快手 - 业务部门/专家面 - 2026年4月10日
    
- **🎯 面试核心特征：** 纯工业级视角的“降维打击”。极度看重**数据驱动（Metrics-driven）意识**与**前沿架构演进视野**，对初创/Demo 级项目进行了极限施压。
### 🔥 核心拷问 1：上下文压缩策略缺乏“数据量化指标”

- **狙击点：** 当被问及上下文优化的效果时，回答“主要依靠产品经理的反馈”。这在大厂是极其忌讳的，大厂工程演进必须由客观数据指标驱动。
    
- **❌ 我的原始漏洞：** 缺乏“评价体系”意识，停留在“能跑通、感觉还行”的学生思维。
    
- **✅ 迭代后的“大厂级”解法（指标驱动）：**
    
    - **性能指标监控：** 建立大盘监控，量化引入压缩策略前后的 **平均 Token 消耗下降率** 与 **首包延迟（TTFB）的波动方差**。
        
    - **业务指标（自动化评测）：** 建立包含核心 Ground Truth 的测试集，每次修改摘要逻辑后，进行自动化 RAG 召回测试，计算 **召回命中率（Hit Rate）**，用客观数据指导迭代。
### 🔥 核心拷问 2：流式音频技术栈“太老旧”

- **狙击点：** 简历中的双向流式语音后端，被面试官指出架构老旧、有更好的替代方案。
    
- **❌ 我的原始漏洞：** 使用的是传统的 **“级联架构（ASR -> LLM文本 -> TTS）”**。这种方案存在物理延迟极限，且会丢失情绪、语气词等真实表现力。
    
- **✅ 迭代后的“大厂级”解法（前沿演进视野）：**
    
    - **认知对齐：** 承认级联架构的延迟天花板。
        
    - **演进方向：** 下一代极致拟人与极低延迟（<300ms）的架构，必须演进到 **端到端原生音频流模型（End-to-End Audio Native）**，直接让大模型接收并输出音频，跳过中间文本转换。
