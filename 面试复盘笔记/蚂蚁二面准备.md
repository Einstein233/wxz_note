## 问题1：OpenClaw学习原理与记忆管理技术

### 什么是OpenClaw？
OpenClaw 是一个 CLI 驱动的智能体(agent)框架，主要功能包括：
- 管理本地网关设置
- 模型认证
- 技能安装
- 多通道消息路由（Telegram、Discord等）

### 学习原理与记忆管理技术
OpenClaw 的记忆管理核心是通过**模型回退链(Fallback Chain)**来实现的：

1. **模型回退链**：按顺序尝试备选模型
   - 当主模型配额用尽、触发速率限制或其他失败时，自动切换到备选模型
   - 可以添加轻量级备选模型（如gpt-4-mini）来降低内存/延迟成本
   - 也可以移除或清除回退链

2. **模型别名(Aliases)**：提供稳定的模型标识，方便切换使用

3. **调试覆盖(Debug Overrides)**：运行时限制（如maxTokens）直接控制内存消耗

关键命令：
- `openclaw models fallbacks list` - 查看回退链
- `openclaw models fallbacks add <model>` - 添加备选模型
- `openclaw models status` - 查看当前模型状态

## 问题2：Harness是什么意思？什么概念？

**Harness（测试 harness）** 是软件测试中的核心概念，指：

### 定义
一个软件工具和测试数据的集合，用于在受控条件下测试应用程序。

### 包含组件
- **测试脚本**：自动化测试用例
- **测试数据**：用于验证的输入数据
- **桩程序(Stubs)**：模拟未完成的模块
- **驱动程序(Drivers)**：模拟调用被测模块的上层组件
- **模拟器**：模拟不可用的外部依赖

### 主要作用
1. **解耦依赖**：在集成测试中，即使某些模块尚未完成，也可以通过模拟来验证组件间的交互
2. **早期测试**：可以在完整系统准备好之前就进行测试
3. **自动化执行**：自动化测试运行、结果验证和报告生成

### 典型应用场景
- 单元测试和集成测试
- API尚未开发完成时的测试
- 外部服务不可用时的测试

简单来说，harness就是一套"测试骨架/环境"，让测试可以在不完整的环境中也能可靠运行。

## 实现一个CLI智能体？

如何确认权限：
1、目标：让agent可以自动完成任务，但是默认不信任它。尽量避免误删、越权、误操作；
设计目标：1、环境隔离：Agent不能直接碰宿主机和核心数据。2、权限最小化：Agent只拿到完成当前任务所必须的权限；3、高风险可拦截，像删除、覆盖、外部写操作，不能无脑执行。4、可审计可回滚。除了问题知道是谁、什么时候做的，需要可以恢复；
**整体结构**：
用户请求
  ↓
Agent Orchestrator（任务编排层）
  ↓
Planner / ReAct Loop（规划执行层）
  ↓
Policy Engine（权限与风险决策层）
  ↓
Tool Gateway（工具网关层）
  ↓
Sandbox Runtime（隔离执行层）
  ↓
Data / External Services（数据库、Git、文件、API）

关键是agent不能直接调用工具，必须经过policy engine+tool gateway
1、Agent orchestrator：总控层
维护一次任务的状态、保持步骤历史、控制loop次数、token、时间
2、Planner：智能层
理解用户目标，拆分子任务、决定下一步动作，根据observation调整计划
3、Policy Engine：权限与风险决策层
判断这个动作能不能执行、要不要确认、需不需要降权、要不要拒绝。
看身份类型、看任务类型、看资源类型、看动作风险等级
4、toolgateway：工具网关层
所有工具能力统一从一个网关出去，不能让 Agent 直接拿原生系统能力。
5、Sandbox Runtime：隔离执行层
每个任务一个独立 runtime，独立容器 独立工作目录 独立临时卷 任务结束自动销毁。
6、数据与外部系统访问层
数据库访问，只给只读账号，禁止一些操作，限制查询时间，限制返回行数
对于git仓库，只改副本仓库，生成patch/diff，push前人工确认

### harness是什么意思？什么概念？
LLM的本质是：黑箱、不稳定、有幻觉、可能乱执行、输出不可预测
因此需要：  用户 → Harness → LLM / Agent → Harness → 结果
harness的作用就是：控制agent可以做什么，限制不可以做什么，观察做了什么，评估做的好不好；
**Harness 工程到底包含什么？**
1、控制
决定系统如何运行，包括执行流程、step限制、timeout、retry机制
2、约束
限制能力边界，包括权限控制，工具白名单，SQL限制，文件访问控制
3、调度
任务如何推进，包括多步骤执行，子任务拆解，状态流转
4、观测
记录系统行为，包括日志、每一步ation、tool调用、中间结果；
5、评估
判断系统好不好，包括正确率，成功率，任务完成度，cost

深一层理解：是agent的操作系统、安全边界、评估系统
**harness的实现形式：**
1、代码级流程控制
2、规则引擎、策略层，不把所有限制都写在业务代码里，而是单独抽出一层出来，哪些工具可以调用，那些路径可以访问
3、中间件、网关式控制，LLM不直接碰底层工具，而是先走一个中间层。
例如toolgateway，权限检查，参数检查，审计记录，真正工具。
统一拦截、统一日志，统一鉴权更加安全
很常见，例如不让模型直接执行shell，而是只能调用一些工具；
4、状态机/工作流引擎
任务比较复杂时，单纯 while loop 就不够优雅了。
这时候 harness 会做成：
- 有状态流转
- 有节点编排
- 有失败分支
- 有人工插入点
5、沙箱 / 运行时隔离
环境隔离，即使做错了，也可以限制在一个小范围内；
6、观测与审计系统
记录它做了什么，方便分析、debug、追责。
常见实现
- action log
- tool call log
- SQL log
- 文件 diff
- token 用量
- 每一步 reasoning summary
7、评估 Harness
**批量测试系统能力**。
- 给 agent 一批任务集
- 自动跑
- 自动打分
- 比较不同 prompt / model / 策略效果
8、人工在环（Human-in-the-loop）
高风险动作不让模型自动完成，而是插入人工确认。
